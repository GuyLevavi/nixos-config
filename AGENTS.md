# AGENTS.md

Guidance for agentic coding agents operating in this NixOS configuration repository.

## Repository Overview

This is a NixOS flake-based system + home-manager configuration for an Intel desktop (`nixbox`)
running Hyprland on Wayland. The primary user is `gl`.

```
flake.nix                    # entry point — pins nixpkgs (unstable), home-manager, catppuccin/nix
home.nix                     # thin wrapper: imports ./home/gui.nix
home/
  base.nix                   # headless home-manager: shell, neovim, git, CLI tools (also used for WSL)
  gui.nix                    # GUI layer: imports base.nix, adds Hyprland + graphical apps
hosts/nixbox/
  configuration.nix          # system config (boot, hardware, services, users, fonts)
  hardware-configuration.nix # generated by nixos-generate-config — do NOT edit
config/                      # dotfiles read/symlinked at build time
  nvim/                      # LazyVim config (symlinked to ~/.config/nvim)
  hypr/hyprland.conf         # Hyprland config (loaded via extraConfig in gui.nix)
DECISIONS.md                 # rationale for every architectural decision — read before changing anything
```

## Commands

All commands assume nushell unless noted. Run from `/etc/nixos`.

```nushell
rb       # sudo nixos-rebuild switch --flake /etc/nixos#nixbox  — apply all changes
update   # sudo nix flake update /etc/nixos  — bump flake.lock, then rebuild
gcold    # sudo nix-collect-garbage --delete-older-than 14d
nsh ripgrep fd  # temporary nix-shell with packages (nix-shell -p <pkgs>)
```

```bash
# Debug a failed rebuild (run in bash, not nushell):
sudo nixos-rebuild switch --flake /etc/nixos#nixbox --show-trace

# Roll back to the previous generation:
sudo nixos-rebuild switch --rollback

# List generations:
sudo nix-env --list-generations --profile /nix/var/nix/profiles/system

# Evaluate/check a specific file without rebuilding:
nix eval --file /etc/nixos/flake.nix
```

## Build / Lint / Test

**There is no CI pipeline, no test suite, and no pre-commit hooks.** The only validation
mechanism is a successful `nixos-rebuild switch`.

| Task | Command |
|---|---|
| Apply changes (full rebuild) | `rb` (nushell alias) or `sudo nixos-rebuild switch --flake /etc/nixos#nixbox` |
| Debug build failures | add `--show-trace` to the rebuild command |
| Format a Nix file | `nixpkgs-fmt <file.nix>` |
| Format a Lua file | `stylua <file.lua>` (config: `config/nvim/stylua.toml`) |
| Format Python | `black <file.py>` |
| Lint Nix (dead code) | `deadnix <file.nix>` |
| Lint Nix (best practices) | `statix check /etc/nixos` |
| Check a standalone home-manager config | `home-manager build --flake /etc/nixos#wsl` |

All formatters/linters (`nixpkgs-fmt`, `stylua`, `black`, `deadnix`, `statix`) are available in
the environment via `programs.neovim.extraPackages`. To run them ad-hoc outside the editor use
`nix run nixpkgs#nixpkgs-fmt -- <file>` if not in PATH.

## Architecture Constraints

### System vs User split
- **System-level** (boot, hardware, services, users, fonts): `hosts/nixbox/configuration.nix`
- **User-level** (programs, dotfiles, theming): `home/base.nix` and `home/gui.nix`
- **Headless/WSL-safe** config: `home/base.nix` (no graphical dependencies)
- **GUI-only** config: `home/gui.nix` (imports `base.nix`; add graphical apps here)

### Adding packages — where to put them
| What | Where |
|---|---|
| User CLI app | `home.packages` in `home/base.nix` |
| User GUI app | `home.packages` in `home/gui.nix` |
| App needing catppuccin theming | `programs.<name>.enable = true` in the appropriate home file |
| System daemon / hardware driver | `hosts/nixbox/configuration.nix` |
| New LSP server or formatter | `programs.neovim.extraPackages` in `home/base.nix` |

### Catppuccin theming — modules, not home.file
All theming goes through `catppuccin/nix`. Flavor `mocha`, accent `mauve` are set once.

**CRITICAL:** A catppuccin module only fires when `programs.<name>.enable = true` (or `services.<name>.enable = true`) is also set.
Installing via `home.packages` or `home.file.source` bypasses the hook — theming will not apply.
Never use `home.file` to drop in a catppuccin theme file manually.

**Modules active in gui.nix:** `hyprland`, `waybar`, `hyprlock`, `swaync`, `rofi`, `kitty`, `gtk.icon`

### LSP servers — Nix provides, Mason does not install
LSP servers and formatters live in `programs.neovim.extraPackages`. Mason's `ensure_installed`
is set to `{}` in `config/nvim/lua/plugins/lsp.lua`. Do **not** run `:MasonInstall` — FHS
binaries fail on NixOS. `:Mason` showing servers as "not installed" is cosmetic only.

### Shell setup
- Bash is the PAM login shell (required for systemd/emergency mode)
- On interactive start, `.bashrc` execs nushell (guarded with `[[ -z "$IN_NIX_SHELL" ]]`)
- Inside `nix-shell`, bash does NOT auto-exec nushell
- Nushell aliases go in `configFile.text` (config.nu), NOT in `envFile.text` (env.nu)

### stateVersion
`system.stateVersion` and `home.stateVersion` are both `"25.05"`. **Do not change them.**

## Code Style

### Nix formatting
- **2-space indentation** — no tabs
- `nixpkgs-fmt` is the canonical formatter; run it before committing any `.nix` file
- Section headers use em-dash box-drawing lines:
  ```nix
  # ── Section name ──────────────────────────────────────────────────────────
  ```
- Use `with pkgs;` for package list blocks to avoid repeating `pkgs.`
- Inline comments explain *why*, not just *what*
- No trailing commas (Nix syntax does not require them in lists/attrsets)
- Selective attribute alignment for related boolean flags (e.g. pipewire options)

### Nix patterns
- Use `lib.mkForce` when a `gui.nix` value must override a `base.nix` default
- Use `lib.hm.dag.entryBefore` when activation script ordering matters
- Use `lib.concatStrings` to compose multi-part string values (e.g. starship format)
- Inline multi-line config strings (nushell, bash) directly in Nix using `''...''` strings —
  do not create separate dotfiles unless the tool requires a standalone file
- For programs with a HM module, always use `programs.<name>` rather than raw `home.file`

### Lua (Neovim config)
- Formatter: `stylua` with settings from `config/nvim/stylua.toml`
  - `indent_type = "Spaces"`, `indent_width = 2`, `column_width = 120`
- Follow LazyVim plugin spec conventions (`return { "author/plugin", opts = { ... } }`)
- Disable plugins with `enabled = false` inside the spec, not by deleting files

### Naming conventions
- Machine hostname: `nixbox` — used as the `nixosConfigurations` key and `--flake` target
- Primary user: `gl`
- Nix files: `lowercase-with-hyphens.nix`
- `homeConfigurations.wsl` — standalone home-manager target for WSL/headless deployments
- Avoid abbreviations in option names; be explicit (`extraPackages`, not `pkgs`)

## Error Handling / Debugging

- Always rebuild with `--show-trace` when diagnosing attribute errors or import failures
- When a module option does not exist, check `nixpkgs` version: this repo tracks `nixos-unstable`
- If `nixos-rebuild switch` fails mid-way, the previous generation is still bootable; use
  `sudo nixos-rebuild switch --rollback` to revert
- D-Bus cannot be hot-swapped: if changing `services.dbus.implementation`, use
  `nixos-rebuild boot` then reboot instead of `switch`
- The Podman socket must be explicitly started: `systemctl --user enable --now podman.socket`
- `DOCKER_HOST` is set to the rootless Podman socket — do not install Docker Engine

## Important Notes

- `flake.lock` is generated — do not edit by hand; use `update` alias to bump
- `hosts/nixbox/hardware-configuration.nix` is generated — do not edit by hand
- `config/nvim/` is symlinked (not copied) to `~/.config/nvim` at activation time
- Chrome Catppuccin theme must be installed manually (browser extensions cannot be declared)
- Zoxide uses `--cmd cd` to shadow nushell's built-in `cd` — no alias needed
- Before making any architectural change, read `DECISIONS.md` for prior rationale
